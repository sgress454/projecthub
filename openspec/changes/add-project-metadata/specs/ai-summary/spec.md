## ADDED Requirements

### Requirement: AI-generated project summary

Each project SHALL have a cached AI-generated summary that describes the current state of the project in 2-3 sentences. The summary is generated via the `claude` CLI and persisted on the project.

#### Scenario: Summary includes project goal when OpenSpec is linked

- **GIVEN** a project has an `openspecChange` set and the change has a `proposal.md`
- **WHEN** the summary is generated
- **THEN** the summary includes a brief description of the project's goal derived from the proposal

#### Scenario: Summary includes PR status

- **GIVEN** a project has open PRs with unresolved reviewer comments
- **WHEN** the summary is generated
- **THEN** the summary mentions the open PRs and highlights unresolved comment counts

#### Scenario: Summary includes recent git activity

- **GIVEN** a project has a `path` set and the directory is a git repository
- **WHEN** the summary is generated
- **THEN** the context sent to Claude includes recent git log output for the project's branch

#### Scenario: Summary includes OpenSpec task progress

- **GIVEN** a project has an `openspecChange` set and the change has a `tasks.md`
- **WHEN** the summary is generated
- **THEN** the summary reflects task completion progress (e.g., "3 of 5 tasks done")

#### Scenario: Summary with no context available

- **GIVEN** a project has no linked issues, PRs, OpenSpec change, or git history
- **WHEN** the summary would be generated
- **THEN** the summary is set to nil and the submenu shows the fallback message

### Requirement: Summary regeneration triggers

The summary SHALL be regenerated when the project's underlying data changes, not on a fixed timer.

#### Scenario: Regenerate after GitHub sync finds changes

- **WHEN** a GitHub sync cycle completes and the PR list or comment counts have changed for a project
- **THEN** a summary regeneration is triggered for that project

#### Scenario: Regenerate after metadata edit

- **WHEN** the user adds or removes an issue, PR, link, or changes the OpenSpec association
- **THEN** a summary regeneration is triggered for that project

#### Scenario: Regenerate after git activity

- **WHEN** a file change is detected in the project's git repository that results in a new commit on the current branch
- **THEN** a summary regeneration is triggered for that project

#### Scenario: Debouncing multiple triggers

- **WHEN** multiple regeneration triggers fire within 5 seconds for the same project
- **THEN** only one summary generation is performed, after the 5-second debounce window

### Requirement: Summary generation via Claude CLI

The summary SHALL be generated by invoking `claude -p` with a prompt and gathered context.

#### Scenario: Claude CLI invoked with project context

- **WHEN** a summary is generated for a project
- **THEN** the app invokes `claude -p` with a system prompt requesting a 2-3 sentence project status summary, passing the gathered context (git log, PR info, issue info, OpenSpec docs) as input

#### Scenario: Summary cached on project

- **WHEN** Claude CLI returns a summary
- **THEN** the result is stored in `project.summary` and persisted to disk

#### Scenario: Claude CLI failure

- **WHEN** `claude -p` exits non-zero, times out, or is not installed
- **THEN** `project.summary` is set to nil and a warning is logged

#### Scenario: Summary generation is asynchronous

- **WHEN** a summary generation is triggered
- **THEN** the generation runs in the background and the menu bar continues to display the previously cached summary (or fallback) until the new one is ready

### Requirement: Graceful degradation without Claude CLI

The app SHALL function normally when the `claude` CLI is not available. Summary generation is silently disabled.

#### Scenario: Claude CLI not installed

- **WHEN** the app attempts to generate a summary and `claude` is not found on PATH
- **THEN** summary generation is skipped, `project.summary` remains nil, and the submenu shows the fallback message
